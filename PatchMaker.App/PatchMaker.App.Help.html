PatchMaker V1.2 Help

This application is an attempt at building some UI to make creating Sitecore config patch files easier. It's an experiment hacked together by Jeremy Davis, so it's probably got a load of rough edges. When you find issues, as you inevitably will, you can open bugs on GitHub. I'll have a think about whatever gets raised there - though I can't promise to fix everything.

Loading base XML

The first step to creating a patch is to get some source XML to work from. This is the XML you want to apply patches to. You might open config files from your local Sitecore instance, or you might save the ShowConfig.aspx page from your web browser and open that.

  • Click the "File" menu, and select "Load XML".
  • Use the dialog to find and select a valid Sitecore config XML file.

The XML will open in the left-hand side of the main window. You can expand the tree by left clicking on the "+" icons.

Planning patches

To start creating patches, you navigate to the element you want to work on. You can scroll through the XML tree, expanding stuff to find the node you want manually. Or you can search for it. Type a fragment into the text box above the tree view, and hit enter or the search icon:

The search does a case-insensitive "contains" on the text of the tree nodes - so will match element names or attribute names/values. Matched nodes are highlighted in red. The tree will expand so all matched nodes are visible. If you want to clear the highlight, search for an empty string. If you want to collapse the tree, click the "collapse" icon to the right of the search one.

Once you've found the correct element, right-click it for the list of patching options you can use.

  • Changing attributes:

    You can use both the "set:attribute" and "patch:attribute" methods from the same dialog.

    The tree in these dialogs serves as a reminder of what path you're working on in the XML document.

    The "Element" field specifies the xPath to find the element you want to modify. The default is the exact path to the element you right-clicked to open this dialog - but you can edit this if you require.

    Changing the "Type" field lets you pick which approach to patching elements you want to use - either "patch:attribute" or "set:attribute".

    The "Name" and "Value" specify the attribute you're changing. The name for the attribute is a required field. You can type whatever you need in here (as long as they're valid for XML attributes) or you can use the "Default" dropdown to pick one of the pre-existing attributes for the element you're changing and then click the "<<" to populate the fields before you do your editing.

  • Deleting elements:

    The "xPath" field here specifies which element you want to delete. It defaults to the exact path of the element you right-clicked. You can edit it if necessary.

  • Inserting elements:

    The field for "Parent" specifies the element that will be the parent of your new node. It defaults to the XPath for the parent of the element you right-clicked. The "Order" fields allow you to specify where your new element will go. The textbox specifies the relative xPath for the element you're basing your insertion around, and the dropdown allows you to pick if it's inserted before or after whatever the XPath matches.

    The "Xml" field lets you type the XML you want inserted. This needs to be valid, and you will get validation messages if it is not. When invalid XML is entered here you can only cancel this dialog until you fix it. You cannot change field or click "Ok" until the XML error is solved.

  • Replacing elements:

    As for inserting, the "Parent" field species the parent of the element you're changing. The "Element" field is the xPath for the specific element being replaced. And the "Xml" field allows you to enter the new XML as for inserting.

  • Adding a new child:

    You can only use this option if you selected a "leaf" node in the XML tree. It's not actually a patch directive, it's just a way to add a new element to the patch document so it appears as a child of whatever you selected. The "Parent" field species the parent of the element you're going to make a new child for. And as above, the the "Xml" field is the new element(s) to insert under the parent.

Adjusting patches

When you create a patch, it gets added to the list in the right-hand pane of the main window.

The order here specifies the order that patches will be built in when you generate a patch file. You can right-click the patch entries here to move them up and down, to change this order. You can also delete them if you wish.

Double clicking a patch here (or right-click and "Edit") will re-open the patch-creation dialog from above, and allow you to modify your settings.

Generating the patch file

When you're ready, you can click "File" and "Generate Patches" to see what your patch file looks like:

The patch XML is generated from the data you entered - but you can modify it if you wish. As above, it must be valid XML to save it.

If you want to see what the patch file will do to your original XML document, click the "Preview" button:

That will use Sitecore's own XML patching engine to apply your patch file to the XML you originally opened, and it will display the result for you to examine.

** NOTE: ** For licensing reasons the Sitecore DLLs can't be shipped with this pre-built releases of this app. You need to drop "Sitecore.Kernel.dll" into the same folder as the "PatchMaker.App.exe" file, before you start the app for this feature to be enabled. The preview button will be disabled if this DLL is not present. The code was built against Sitecore v8.2 - so you may also need to add a version redirect to the application's config file if you wish to use a different version of Sitecore. Or grab the source and change the NuGet reference to Sitecore.

Click the "Save" button to write the patch XML to a disk file. You can deploy it to your server now...
