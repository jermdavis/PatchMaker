PatchMaker V1.6.0 Help

This application is an attempt at building some UI to make creating Sitecore config patch files easier. It's an experiment hacked together by Jeremy Davis, so it's probably got a load of rough edges. When you find issues, as you inevitably will, you can open bugs on GitHub. I'll have a think about whatever gets raised there - though I can't promise to fix everything.

Loading base XML

The first step to creating a patch is to get some source XML to work from. This is the XML you want to apply patches to. You might open config files from your local Sitecore instance, or you might save the ShowConfig.aspx page from your web browser and open that.

  • Click the "File" menu, and select "Load XML".
  • Use the dialog to find and select a valid Sitecore config XML file.

The XML will open in the left-hand side of the main window. You can expand the tree by left clicking on the "+" icons.

Planning patches

To start creating patches, you navigate to the element you want to work on. You can scroll through the XML tree, expanding stuff to find the node you want manually. Or you can search for it. Type a fragment into the text box above the tree view, and hit enter or the search icon:

The search does a case-insensitive "contains" on the text of the tree nodes - so will match element names or attribute names/values. Matched nodes are highlighted in red. The tree will expand so all matched nodes are visible. If you want to clear the highlight, search for an empty string. If you want to collapse the tree, click the "collapse" icon to the right of the search one.

Once you've found the correct element, right-click it for the list of patching options you can use.

  • Changing attributes:

    You can use both the "set:attribute" and "patch:attribute" methods from the same dialog.

    The tree in these dialogs serves as a reminder of what path you're working on in the XML document.

    The "Element" field specifies the xPath to find the element you want to modify. The default is the exact path to the element you right-clicked to open this dialog - but you can edit this if you require.

    Changing the "Type" field lets you pick which approach to patching elements you want to use - either "patch:attribute" or "set:attribute".

    The "Name" and "Value" specify the attribute you're changing. The name for the attribute is a required field. You can type whatever you need in here (as long as they're valid for XML attributes) or you can use the "Default" dropdown to pick one of the pre-existing attributes for the element you're changing and then click the "<<" to populate the fields before you do your editing.

  • Deleting elements:

    The "xPath" field here specifies which element you want to delete. It defaults to the exact path of the element you right-clicked. You can edit it if necessary.

  • Inserting elements:

    The field for "Parent" specifies the element that will be the parent of your new node. It defaults to the XPath for the parent of the element you right-clicked. The "Order" fields allow you to specify where your new element will go. The textbox specifies the relative xPath for the element you're basing your insertion around, and the dropdown allows you to pick if it's inserted before or after whatever the XPath matches.

    The "Xml" field lets you type the XML you want inserted. This needs to be valid, and you will get validation messages if it is not. When invalid XML is entered here you can only cancel this dialog until you fix it. You cannot change field or click "Ok" until the XML error is solved.

  • Replacing elements:

    As for inserting, the "Parent" field species the parent of the element you're changing. The "Element" field is the xPath for the specific element being replaced. And the "Xml" field allows you to enter the new XML as for inserting.

    The new element textbox here has the same namespace behaviour as the insert operation above.

  • Adding a new child:

    You can only use this option if you selected a "leaf" node in the XML tree. (Elsewhere you probably want to use "insert" so you can specify the order for where your new node appears in the tree.) It's not actually a patch directive, it's just a way to add a new element to the patch document so it appears as a child of whatever you selected. The "Parent" field species the parent of the element you're going to make a new child for. And as above, the the "Xml" field is the new element(s) to insert under the parent.

    The new element textbox here has the same namespace behaviour as the insert operation above.

Rule based config

If you're working with Sitecore v9+ and you need to add rules like role:requires attributes, then you can make use of the "Rules" button visible at the bottom of each of the patch creation forms above. Clicking that button allows you to manage the set of rules (and their namespaces) that are added to each patch entry.

By default a patch has zero rules attached. When you click the "Rules" button, you'll see an empty management dialog:

Click "Add" to create a new rule entry. For each entry you need to specify a namespace, an action and a value. There are some default choices for namespace, which you can pick from the combo box dropdown. Or you can type your own URI here. Note that the code assumes the last part of the URI is the namespace name, so you need to follow Sitecore's "http://www.sitecore.net/xmlconfig/NAME" pattern. The action is probably "require" in most cases. And the value is whatever you need matched like "Standalone", or "Solr":

Once you've entered your values, you need to click "Save" before proceeding, to make sure they're recorded. If you've changed the current selection you won't be able to proceed without saving or cancelling. When you close the dialog, the "Rules" button will update to show how many rules are defined for that patch. And when you generate the patches, the relevant namespaces will be added, as well as the rule based config attributes.

When you preview the patches (see below) you can set the relevant rule dara to verify the behaviour of your choices here.

Adjusting patches

When you create a patch, it gets added to the list in the right-hand pane of the main window.

The order here specifies the order that patches will be built in when you generate a patch file. You can right-click the patch entries here to move them up and down, to change this order. You can also delete them if you wish.

Generating the patch file

When you're ready, you can click "File" and "Generate Patches" to see what your patch file looks like:

(This menu option isn't available if you've not added any patches - it won't generate a "nothing" patch for you) The patch XML is generated from the data you entered - but you can modify it if you wish. As above, if you make changes it must be valid XML to save it.

Once you're happy with your patch, and it's valid, you can click the "Save" button to write the patch XML to a disk file. You can deploy it to your server now...

The "Roles" button lets you configure what role(s) you want the patching engine to use when it processes your patch against the source file. If you're not using the "role:require" attribute in your patch you can probably ignore this, but otherwise you can click this button to configure this:

Click "Add to add a new entry to the list. Click "Delete" to remove the currently selected item. If you left-click an item to select it, its values will be added to the textboxes below the list. Type the changes you need into the textboxes, and click "Update" to save your changes. If you're defining multiple values for the "role" entry, use a pipe character to separate them.

If you click "Ok" your changes are remembered. If you click "Cancel" any changes you make are forgotten.

Previewing the result of patching

If you want to see what the patch file will do to your original XML document, click the "Preview" button:

That will use Sitecore's own XML patching engine to apply your patch file to the XML you originally opened, and it will display the result for you to examine. It will attempt to process any role configuration options using the settings you made in the role config dialog above. Any lines which have the patch:source attribute added as part of this preview operation will be highlighted in red. (Though note it highlights lines, not complete XML elements and, kind of obviously, it can't highlight deletions)

You can use the "Next" button to move between patched lines in this view.

If the source XML you've started was itself a patch then this dialog will attempt to sanitise it before trying to process it. Sitecore's patching engine does not like to transform a file which is itself a patch, so you get odd results if this processing doesn't happen first. It will do two things: It will look at any "xmlns" declarations on the root element, and will remove all the attributes in those namespaces from the remaining XML. And secondly, it will remove the "/configuration" root element if it exists, and will promote the child "sitecore" element to be the root. This allows the processing of the source xml with your patch to proceed - but it does mean that the source XML you're processing won't be exactly what you may have expected. You'll be able to see if your source XML has been tweaked like this because it will be marked as "simplified" in the top right of the preview window:

Errors

Despite best efforts it is possible to write patches that are not valid against the source XML you started from. If an error prevents the generation of the patch, then the UI will add a comment into the generated patch XML:

These error comments are always placed at the top of the file. You should correct these issues (usually caused by getting the XPath for the target or parent element to process wrong) before you proceed. They will be ignored by Sitecore's patching process. Note that when an error is raised by patch generation, you cannot preview your patch. These failed patches tend to crash the patch processing engine in Sitecore.

It's also possible to write a patch that generates ok, but Sitecore's patching engine cannot process correctly. You'll see a similar comment in the preview window in this case - though the error messages can be fairly unhelpful things like "null reference" or "namespace not defined" in this case.

** NOTE: ** For licensing reasons the Sitecore DLLs can't be shipped with this pre-built releases of this app. You need to drop "Sitecore.Kernel.dll" into the same folder as the "PatchMaker.App.exe" file, before you start the app for this feature to be enabled. The preview button will be disabled if this DLL is not present. The code was built against Sitecore v9.0 - so you may also need to add a version redirect to the application's config file if you wish to use a newer version of Sitecore. Since the app now makes use of role-based patching APIs, the preview code will not work with earlier versions of this DLL.
